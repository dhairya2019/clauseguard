# Phase 01.1: Harden Skill -- Deep Contract Knowledge, Tests, Docker Validation - Research

**Researched:** 2026-03-22
**Domain:** Contract clause pattern expansion, Indian legal provisions, test harness architecture, Docker validation
**Confidence:** HIGH (clause patterns and Indian law are well-documented; Docker/test harness approach is straightforward)

## Summary

This phase transforms ClauseGuard from a functional prototype (~18 clause patterns, ~5 Indian legal provisions) into a hardened, validated contract analysis system. The work splits into four distinct domains: (1) expanding clause-patterns.md from ~18 to 50+ risky patterns across all categories including new categories like confidentiality, force majeure, auto-renewal, assignment, warranties, and data protection; (2) expanding india-law.md to 20+ legal provisions with real case citations from Delhi HC and Supreme Court; (3) creating 10+ realistic sample contract test fixtures across 5 document types; and (4) building a Docker-based test harness that invokes Claude Code in headless mode (`claude -p`) against each sample contract and validates that expected risk flags appear in the output.

The critical insight is that the test harness does NOT need to call Claude API directly -- it invokes the installed ClauseGuard skill via `claude -p "/clauseguard ./test-contracts/freelance-01.txt"` and validates the output using grep/regex for expected risk markers. This makes the tests deterministic at the validation layer even though the LLM output varies.

**Primary recommendation:** Expand knowledge files first (clause-patterns.md, india-law.md), then create test fixtures with embedded known-risk clauses, then build the bash/Docker test harness that validates the skill's output against expected flags.

### Pattern 1: Headless Skill Invocation for Testing

**What:** Run ClauseGuard skill via `claude -p` in non-interactive mode against test contracts
**When to use:** Every test run
**Example:**
```bash
# Invoke skill against a test contract
OUTPUT=$(claude -p "/clauseguard ./tests/contracts/freelance-01-red.txt" \
  --output-format text \
  --max-turns 1 \
  2>/dev/null)

# Validate expected markers appear
echo "$OUTPUT" | grep -q "HIGH" || fail "Missing HIGH risk flag"
echo "$OUTPUT" | grep -q "S.27" || fail "Missing S.27 reference"
echo "$OUTPUT" | grep -q "Disclaimer" || fail "Missing disclaimer"
```

### Pattern 2: Expected Marker Validation

**What:** Each test fixture has an `.expected` file listing markers that MUST appear in output
**When to use:** For every contract fixture
**Example expected file format:**
```
# freelance-01-red.expected
# Lines starting with # are comments
# Each non-comment line is a pattern that MUST appear in output (grep -i)
MUST:HIGH
MUST:payment
MUST:IP
MUST:termination
MUST:Disclaimer
MUST:not legal advice
SHOULD:S.27
SHOULD:non-compete
MUSTNOT:GREEN document
```

### Pattern 3: Docker Test Environment

**What:** Dockerfile that installs Claude Code CLI and runs test suite
**When to use:** CI/CD and full validation runs
**Example:**
```dockerfile
FROM node:22-slim
RUN npm install -g @anthropic-ai/claude-code
WORKDIR /app
COPY . .
# API key must be passed at runtime
CMD ["bash", "tests/run-tests.sh"]
```

### Anti-Patterns to Avoid

- **Exact string matching on LLM output:** LLM wording varies. Use keyword/pattern matching, not exact string comparison.
- **Testing for specific clause numbering:** The skill may number clauses differently each run. Test for presence of risk flags, not ordering.
- **Running tests without API key budget limits:** Each test invocation costs API tokens. Use `--max-turns 1` to limit cost.
- **Hardcoding expected output:** The goal is to verify risk detection, not exact wording. Use MUST/SHOULD/MUSTNOT markers.

## Clause Pattern Expansion

### Current State (clause-patterns.md)

The existing file has ~18 patterns across 6 Tier 1 categories: Payment, IP, Termination, Liability, Scope, Non-Compete. Each has HIGH/MEDIUM/LOW triggers with detection keywords.

### Expansion Target: 50+ Patterns Across 12 Categories

**Existing 6 categories (expand each with 2-3 additional patterns):**

#### 1. Payment Terms (current: 12 patterns, add 4)
- HIGH: Pay-when-paid / pay-if-paid clauses ("payment contingent on client receiving payment from end client")
- HIGH: Right to offset / set-off without notice ("client may deduct from payments any amounts owed")
- MEDIUM: Payment only for "accepted" work with no acceptance timeline
- MEDIUM: Retention / holdback clauses (10-20% withheld until project end)

#### 2. IP Ownership (current: 11 patterns, add 4)
- HIGH: Assignment of "ideas" and "concepts" (not just deliverables)
- HIGH: IP assignment survives termination with no payment obligation
- MEDIUM: "Background IP" not defined (ambiguous pre-existing IP boundary)
- MEDIUM: License-back restricted to single named project only

#### 3. Termination (current: 9 patterns, add 4)
- HIGH: Termination for insolvency (may be unenforceable under Insolvency and Bankruptcy Code)
- HIGH: Forfeiture of all payments on termination for cause
- MEDIUM: Automatic termination on change of control
- MEDIUM: Survival clause covering unreasonable scope of obligations

#### 4. Liability & Indemnification (current: 9 patterns, add 4)
- HIGH: Indemnification for third-party IP claims where contractor has no control over client's use
- HIGH: "Gross negligence" and "willful misconduct" carve-outs from liability cap (voids the cap effectively)
- MEDIUM: Indemnification for regulatory fines (potentially uncapped)
- MEDIUM: Insurance requirements exceeding reasonable coverage for contract value

#### 5. Scope of Work (current: 9 patterns, add 3)
- HIGH: "Reasonable requests" or "as reasonably required" language (unlimited scope)
- MEDIUM: Acceptance by silence (deliverable deemed accepted if no response in N days -- risk depends on N)
- MEDIUM: Warranty period for deliverables exceeding 90 days

#### 6. Non-Compete (current: 9 patterns, add 2)
- HIGH: Non-compete disguised as "exclusivity" or "dedicated resources" clause
- MEDIUM: Non-compete tied to confidentiality (conflating two distinct restrictions)

**New categories to add (6 new categories, ~3-4 patterns each):**

#### 7. Confidentiality / NDA (new -- 5 patterns)
- HIGH: Perpetual confidentiality obligation with no sunset clause
- HIGH: Definition includes publicly available information or independently developed work
- HIGH: Confidentiality breach triggers liquidated damages (often disproportionate)
- MEDIUM: No carve-out for legally compelled disclosure (court orders, regulatory requirements)
- MEDIUM: One-way NDA when both parties share information

#### 8. Auto-Renewal & Contract Duration (new -- 4 patterns)
- HIGH: Auto-renewal with 90+ day advance notice requirement to cancel
- HIGH: Auto-renewal with price escalation clause (unspecified or uncapped increase)
- MEDIUM: Evergreen contract with no maximum duration
- MEDIUM: Lock-in period with early termination penalty

#### 9. Assignment & Subcontracting (new -- 4 patterns)
- HIGH: Client can assign contract to any entity without consent
- HIGH: Assignment includes assignment of obligations (not just rights) without consent
- MEDIUM: No-subcontracting clause for complex deliverables
- LOW: Mutual consent required for assignment

#### 10. Force Majeure (new -- 3 patterns)
- MEDIUM: No force majeure clause in long-term contract (>6 months)
- MEDIUM: Force majeure only protects one party
- MEDIUM: Force majeure definition excludes pandemic/epidemic/government action

#### 11. Data Protection & Privacy (new -- 4 patterns)
- HIGH: No data processing obligations defined when handling personal data
- HIGH: Contractor liable for client's data practices / data breaches
- MEDIUM: No breach notification timeline specified
- MEDIUM: Cross-border data transfer without compliance mechanism

#### 12. Warranties & Representations (new -- 4 patterns)
- HIGH: Unlimited warranty period ("in perpetuity")
- HIGH: Warranty of fitness for purpose (vs. workmanlike effort)
- MEDIUM: Warranty that work does not infringe any IP (impossible to guarantee absolutely)
- MEDIUM: Representation of "exclusive" skills or qualifications not actually required

**Total after expansion: 50+ patterns (current ~18 expanded + ~30 new)**

### Edge Case Patterns (HARDEN-06)

These are patterns that are hard to detect because of how they are embedded in the contract text:

#### 1. Multi-Clause Interaction Risks
- Non-compete clause + IP assignment clause together = cannot work in your field AND client owns your tools
- Indemnification + no liability cap = theoretically unlimited financial exposure
- Termination for convenience + no payment for in-progress work + IP assignment on termination = client gets everything, pays nothing

#### 2. Buried Clauses in Boilerplate
- Exclusivity buried in "Definitions" section (defining "Engagement" as exclusive)
- Non-compete buried in "General Provisions" or "Miscellaneous"
- IP assignment buried in "Confidentiality" section
- Indemnification obligations hidden in "Representations and Warranties"

#### 3. Misleading Headings
- "Intellectual Property Protection" that actually assigns all IP away from creator
- "Mutual Obligations" with entirely one-sided terms
- "Standard Terms" containing non-standard, aggressive clauses
- "Flexible Engagement" that actually locks in exclusivity

#### 4. Compound Sentences Hiding Risk
- "Contractor assigns all IP AND agrees not to compete for 2 years" (two risky clauses in one sentence)
- "Payment upon completion AND acceptance at client's sole discretion" (payment trap hidden in conjunction)
- "Work product includes all materials, tools, ideas, concepts, and methodologies" (scope creep through list expansion)

#### 5. Defined Term Manipulation
- "Confidential Information" defined so broadly it includes publicly available data
- "Deliverables" defined to include future work not yet scoped
- "Compensation" defined to exclude certain types of work
- "Termination for Cause" defined so broadly any minor issue qualifies

#### 6. Cross-Reference Traps
- "Subject to Section 12" where Section 12 contains aggressive carve-outs
- "As defined in Exhibit B" where Exhibit B has not been provided
- "In accordance with Company Policy" where policy can change unilaterally

## Test Harness Architecture

### Approach: Headless Claude Code + Pattern Validation

The test harness works in two stages:
1. **Invoke:** Run `claude -p "/clauseguard <contract-file>"` in headless mode
2. **Validate:** Check output against `.expected` marker files using grep

### Key Claude Code CLI Flags

```bash
claude -p "/clauseguard ./path/to/contract.txt"  # Headless invocation
  --output-format text                             # Plain text output (default)
  --max-turns 1                                    # Limit to single response (cost control)
  --no-user-prompt                                 # No interactive prompts (CI-safe)
```

**Prerequisites:**
- Node.js 18+ (recommended 22 LTS)
- `@anthropic-ai/claude-code` installed globally or via npx
- `ANTHROPIC_API_KEY` environment variable set
- ClauseGuard skill installed in `.claude/skills/clauseguard/`

### Test Runner Script Structure

```bash
#!/usr/bin/env bash
# tests/run-tests.sh

PASS=0
FAIL=0
WARN=0
CONTRACT_DIR="./tests/contracts"
EXPECTED_DIR="./tests/expected"
OUTPUT_DIR="./tests/output"

mkdir -p "$OUTPUT_DIR"

for contract in "$CONTRACT_DIR"/*.txt; do
  name=$(basename "$contract" .txt)
  expected="$EXPECTED_DIR/${name}.expected"
  output="$OUTPUT_DIR/${name}.output"

  echo "Testing: $name"

  # Run skill in headless mode
  claude -p "/clauseguard $contract" \
    --output-format text \
    --max-turns 1 \
    > "$output" 2>/dev/null

  # Validate against expected markers
  while IFS= read -r line; do
    [[ "$line" =~ ^#.*$ || -z "$line" ]] && continue
    type="${line%%:*}"
    pattern="${line#*:}"
    case "$type" in
      MUST)
        if ! grep -qi "$pattern" "$output"; then
          echo "  FAIL: MUST pattern not found: $pattern"
          ((FAIL++))
        else
          ((PASS++))
        fi
        ;;
      SHOULD)
        if ! grep -qi "$pattern" "$output"; then
          echo "  WARN: SHOULD pattern not found: $pattern"
          ((WARN++))
        else
          ((PASS++))
        fi
        ;;
      MUSTNOT)
        if grep -qi "$pattern" "$output"; then
          echo "  FAIL: MUSTNOT pattern found: $pattern"
          ((FAIL++))
        else
          ((PASS++))
        fi
        ;;
    esac
  done < "$expected"
done

echo ""
echo "Results: $PASS passed, $FAIL failed, $WARN warnings"
exit $((FAIL > 0 ? 1 : 0))
```

### Cost Considerations

Each test invocation calls the Anthropic API through Claude Code. With ~12 test contracts and `--max-turns 1`:
- Estimated cost: ~$0.50-1.00 per full test suite run (depends on contract length and model)
- Use `--max-turns 1` to prevent multi-turn conversations
- Cache output files during development; only re-run when skill or patterns change

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Contract text generation for fixtures | AI-generated synthetic contracts | Hand-written fixtures with deliberate clause insertion | LLM-generated contracts may miss specific risk patterns; hand-crafted ensures known risks are embedded |
| Complex test assertion framework | Custom assertion library | grep + bash script with MUST/SHOULD/MUSTNOT markers | Simple, zero-dependency, sufficient for text pattern validation |
| LLM output normalization | Custom parser for LLM output | Raw text grep with case-insensitive matching | LLM output format varies; trying to parse structured sections is fragile |
| Indian legal research validation | Custom citation checker | Manual verification of case citations against Indian Kanoon | Legal citations must be accurate; no automated tool can verify Indian case law |

**Key insight:** The test harness validates risk detection, not output formatting. Keep validation simple (grep patterns) rather than trying to parse LLM output structure.

## Common Pitfalls

### Pitfall 1: Over-Specifying Expected Output
**What goes wrong:** Tests break because LLM phrases things slightly differently each run
**Why it happens:** Output is non-deterministic; same clause may be described differently
**How to avoid:** Use keyword/concept markers (MUST:non-compete) not exact phrases (MUST:"This clause contains a non-compete")
**Warning signs:** Tests pass sometimes and fail sometimes with no code changes

### Pitfall 2: Test Fixtures That Are Too Simple
**What goes wrong:** Skill works great on simple contracts but fails on real-world messy ones
**Why it happens:** Real contracts have buried clauses, cross-references, compound sentences
**How to avoid:** Include edge case fixtures (HARDEN-06) with buried and disguised risk patterns
**Warning signs:** 100% pass rate on tests but users report missed risks

### Pitfall 3: Incorrect Indian Legal Citations
**What goes wrong:** Case citations are wrong or outdated; erodes trust with legally-savvy users
**Why it happens:** Training data may have incorrect case details; citations may be hallucinated
**How to avoid:** Every citation must be verified against Indian Kanoon (indiankanoon.org) or official court records
**Warning signs:** Case name/year/court mismatch; citations that cannot be found online

### Pitfall 4: Ignoring Multi-Clause Interactions
**What goes wrong:** Each clause looks acceptable individually but combined creates extreme risk
**Why it happens:** Pattern matching is per-clause; interactions require holistic analysis
**How to avoid:** Add specific edge case patterns for known dangerous combinations (see Edge Case Patterns section)
**Warning signs:** Green individual clauses but RED overall assessment missing

### Pitfall 5: Docker Build Without API Key Handling
**What goes wrong:** Docker image contains hardcoded API key OR tests fail because key is missing
**Why it happens:** API key must be provided at runtime, not build time
**How to avoid:** Pass ANTHROPIC_API_KEY as environment variable at `docker run` time; add .dockerignore for .env files
**Warning signs:** Docker image works locally but fails in CI; API key leaked in image layers
