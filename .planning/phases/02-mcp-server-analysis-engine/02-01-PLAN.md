Create `src/prompts/system-prompt.ts` that extracts and condenses the analysis logic from three source files into a JSON-output-focused system prompt.

Read these three files to extract content:
- `.claude/skills/clauseguard/SKILL.md` (319 lines) — 8-step workflow, risk classification, framing rules, disclaimer
- `.claude/skills/clauseguard/clause-patterns.md` (379 lines) — risk trigger patterns, missing clause checklists, safer alternatives, edge cases
- `.claude/skills/clauseguard/india-law.md` (289 lines) — S.27, S.28, S.25, S.73-74, FEMA, jurisdiction, Copyright Act, Arbitration Act

**Export `buildSystemPrompt()` function** returning a string. The system prompt structure MUST follow Pattern 3 from 02-RESEARCH.md:

```
System prompt structure:
├── Role + output format (instruct Claude to call contract_analysis_result tool)
├── Step 1: Document type identification (from SKILL.md Step 2)
│   - 7 document types with key indicators
├── Step 2: Clause extraction taxonomy (from SKILL.md Step 3)
│   - 6 Tier 1 categories with detection keywords
├── Step 3: Risk classification rules (from SKILL.md Step 4 + clause-patterns.md Section 1)
│   - HIGH/MEDIUM/LOW triggers per category
│   - Include ALL patterns from clause-patterns.md Section 1 (payment, IP, termination, liability, scope, non-compete) — condense into concise IF/THEN rules, not verbose tables
│   - Include edge case patterns from clause-patterns.md Section 4
├── Step 4: India-specific rules (from india-law.md Sections 1-9)
│   - S.27 non-compete with 5-scenario nuance table
│   - S.28 restriction on legal proceedings
│   - S.25 agreement without consideration
│   - S.73-74 penalty provisions
│   - S.29 void for uncertainty
│   - Copyright Act S.17, S.57
│   - FEMA rules (payment channels, currency, repatriation)
│   - Jurisdiction preference hierarchy
│   - Use IF/THEN pseudocode blocks (already structured this way in india-law.md)
├── Step 5: Missing clause checklists per document type (from clause-patterns.md Section 2)
├── Step 6: Safer alternative templates (from clause-patterns.md Section 3)
├── Step 7: Overall risk aggregation formula + escalation triggers
│   - RED document: 2+ HIGH, or 1 HIGH in payment/liability/IP
│   - YELLOW document: 1 HIGH (non-critical) or 3+ MEDIUM
│   - GREEN document: no HIGH, fewer than 3 MEDIUM
│   - Escalation triggers from SKILL.md
├── Framing rules (DO say / DO NOT say from SKILL.md)
└── Disclaimer text (verbatim from SKILL.md)
```

**CRITICAL differences from SKILL.md:**
1. Output format is JSON (tool call), not markdown — instruct Claude to populate the contract_analysis_result tool input
2. The 8-step workflow is implicit (Claude follows internally), not explicit output sections
3. Condense tables into rules — aim for ~4000-6000 tokens total
4. Include key case citations inline (e.g., "Void under S.27 — Percept D'Mark v. Zaheer Khan (2006)")
5. Include detection keywords for each risk pattern so Claude can match against contract text

**Export `buildUserMessage(contractText, analysisType, partyPerspective?)` function** that constructs the user message:
```
Analyze this contract.
Analysis type: {analysisType}
{partyPerspective ? `Perspective: ${partyPerspective}` : ''}

CONTRACT TEXT:
{contractText}
```

**Framing rules MUST be in the system prompt:**
- Always "this clause means..." never "you should..."
- Frame suggestions as "standard market language"
- Never say "illegal" — say "void under S.27" or "unenforceable under S.28"
- Always include indiaNote for clauses triggering India-specific rules

**Disclaimer text (verbatim):**
"This analysis is for informational purposes only and does not constitute legal advice. ClauseGuard is not a law firm. For important contracts or high-value agreements, consult a qualified advocate enrolled with the Bar Council of India."
